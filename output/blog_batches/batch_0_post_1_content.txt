# Amazon Bedrock 완벽 가이드: AIF-C01 시험 합격 전략

## 소개: Amazon Bedrock이란?

**Amazon Bedrock**은 AWS의 완전 관리형 생성형 AI 서비스입니다. 이 서비스는 인프라 관리 없이 다양한 파운데이션 모델(FM)에 접근할 수 있게 해줍니다.

### 핵심 특징

- **완전 관리형 서비스**: 인프라 운영 부담 없음
- **데이터 프라이버시**: 모든 데이터는 AWS 계정 내에만 존재 — 외부 노출 없음
- **종량제 요금**: 사용한 만큼만 비용 청구 (선결제 불필요)
- **표준화된 API**: 모든 모델에 일관된 인터페이스 제공
- **다중 제공사**: AI21 Labs, Anthropic, Cohere, Meta, Stability AI 등 여러 파운데이션 모델 지원

> 💡 **시험 포인트**: AIF-C01 시험에서는 Bedrock의 '완전 관리형' 특성과 '데이터 프라이버시 보장'이 자주 출제됩니다.

## 파운데이션 모델 선택 전략

### 지원되는 모델 종류

Amazon Bedrock은 다음 제공사의 파운데이션 모델을 호스팅합니다:

| 제공사 | 대표 모델 | 특징 |
|--------|---------|------|
| Amazon | Titan | 멀티모달 지원, 미세조정 가능, 비용 효율적 |
| Anthropic | Claude | 가장 큰 컨텍스트 윈도우(200K), 복잡한 분석 최적 |
| Meta | Llama-2 | 대규모 작업, 4K 토큰 컨텍스트, 중간 가격대 |
| Stability AI | Stable Diffusion | 이미지 생성만, 이미지당 과금 |

**중요**: 각 모델은 계정 내에서 독립적으로 격리되며, 모든 데이터는 계정 내에만 유지됩니다.

### 모델 선택 기준

```mermaid
flowchart TD
    A[작업 유형 파악] --> B{텍스트 생성 필요?}
    B -->|Yes| C{컨텍스트 윈도우 크기?}
    C -->|소규모| D[Amazon Titan]
    C -->|대규모| E[Claude]
    B -->|No| F{이미지 생성?}
    F -->|Yes| G[Stable Diffusion]
    F -->|No| H[재고려]
```

모델 선택 시 고려해야 할 주요 요소:

✅ **모델 타입 & 성능**: 다양한 작업에 최적화된 모델
✅ **컨텍스트 윈도우**: 처리할 수 있는 최대 데이터 크기
✅ **비용**: 토큰 기반 요금, 모델마다 상이
✅ **언어 지원**: 대부분 영어 최적화, 한국어 지원도 확인 필요
✅ **멀티모달 지원**: 텍스트, 이미지, 오디오 등 다중 입력 처리 가능 여부

> ⚠️ **주의**: 더 크고 강력한 모델이 항상 최선은 아닙니다. 비용과 성능의 균형을 맞춰 여러 모델을 테스트하세요.

## 파인튜닝(Fine-tuning): 모델 커스터마이징

### 파인튜닝의 목적

Bedrock에서 제공하는 파운데이션 모델을 **자신의 데이터로 학습**시켜 특정 용도에 최적화할 수 있습니다.

### 파인튜닝의 세 가지 방식

#### 1. 명령어 기반 파인튜닝 (Instruction-Based Fine-Tuning)

**목적**: 도메인별 특정 작업 성능 개선

**데이터 형식**:
```
Prompt: AWS란 무엇인가요?
Completion: AWS는 Amazon Web Services의 약자로, 아마존이 제공하는 클라우드 컴퓨팅 플랫폼입니다.
```

**비용**: 가장 저렴 (계산 강도 낮음)
**사용 사례**: 고객 서비스 챗봇, 도메인 특화 질문 응답

#### 2. 지속적 사전학습 (Continued Pre-training)

**목적**: 파운데이션 모델을 특정 분야 전문가로 만들기

**데이터 형식**: 레이블 없는 순수 텍스트 문서
- AWS 전체 문서 학습 → AWS 전문 모델 생성
- 의료 논문 학습 → 의료 전문 모델 생성

**비용**: 더 높음 (계산 강도 높음)
**사용 사례**: 금융 용어 데이터, 의료 정보 처리, 회사 내부 지식 기반

#### 3. 단/다중 턴 대화 미세조정

**목적**: 대화형 상호작용 개선

**데이터 구조**:
```
System: 당신은 고객 서비스 담당자입니다.
User: 지원 시간이 어떻게 되나요?
Assistant: 24시간 365일 지원합니다.
```

> 💡 **시험 포인트**: AIF-C01에서는 세 가지 파인튜닝 방식의 **차이점**과 **비용 효율성**을 비교하는 문제가 출제됩니다.

## 검색 증강 생성(RAG): 실시간 데이터 통합

### RAG가 필요한 이유

파운데이션 모델은 학습 시점의 지식만 보유합니다. 회사 내부 문서나 최신 정보가 필요한 경우 **파인튜닝 없이** 실시간으로 데이터를 제공할 수 있는 기술이 RAG입니다.

**RAG vs 파인튜닝**:

| 구분 | RAG | 파인튜닝 |
|------|-----|----------|
| 데이터 업데이트 | 실시간 | 재학습 필요 |
| 구현 속도 | 즉시 | 시간 소요 |
| 비용 | 중간 | 높음 |
| 데이터 크기 | 대규모 적합 | 소규모 적합 |
| 사용 시기 | 자주 변하는 정보 | 고정된 도메인 지식 |

### RAG 처리 프로세스

```mermaid
sequenceDiagram
    participant User as 사용자
    participant Bedrock as Amazon Bedrock
    participant KB as Knowledge Base
    participant VectorDB as Vector DB

    User->>Bedrock: 질문: "Product Manager는 누구?"
    Bedrock->>VectorDB: 유사 문서 검색
    VectorDB-->>KB: 관련 정보 반환
    KB-->>Bedrock: 검색 결과 + 컨텍스트
    Bedrock->>Bedrock: 증강된 프롬프트로 답변 생성
    Bedrock-->>User: "Product Manager는 Bob입니다."
```

### RAG 실행 단계

1. **데이터 저장**: Amazon S3, Confluence, SharePoint 등에 문서 저장
2. **Knowledge Base 생성**: Bedrock이 자동으로 임베딩 처리
3. **벡터 데이터베이스**: 의미 기반 검색 인덱스 생성
4. **사용자 질문**: 유사 문서 검색 후 컨텍스트와 함께 모델에 전달
5. **답변 생성**: 정보를 바탕으로 정확한 응답 생성

### 지원되는 데이터 소스

- **Amazon S3**: 문서, PDF 등 파일 저장소
- **Confluence**: 팀 협업 문서
- **Microsoft SharePoint**: 엔터프라이즈 콘텐츠 관리
- **Salesforce**: CRM 고객 정보
- **웹페이지**: 공개 웹사이트, 뉴스 피드

> 💡 **시험 포인트**: RAG를 언제 사용할지가 출제 핵심입니다. '빠르게 변하는 데이터' → RAG, '고정된 도메인 지식' → 파인튜닝이 정답입니다.

## 벡터 데이터베이스와 임베딩

### 벡터 임베딩이란?

**임베딩(Embedding)**은 텍스트, 이미지 등의 데이터를 **숫자 벡터**로 변환합니다.

벡터 저장 후 **의미 기반 유사도 검색(Semantic Search)**이 가능해집니다 — 키워드 일치가 아닌 의미로 검색합니다.

### 벡터 데이터베이스 선택

**AWS 네이티브**:

| 서비스 | 특징 | 최적 용도 |
|--------|------|----------|
| Amazon OpenSearch Service | 실시간 검색·분석 | 프로덕션 RAG, 수백만 임베딩 |
| Amazon Aurora | 관계형 DB + 벡터 지원 | 기존 Aurora 환경 |
| Amazon Neptune Analytics | 그래프 DB 서비스 | 그래프 기반 RAG |
| Amazon S3 Vectors | S3 내 벡터 저장 | 비용 최적화 시나리오 |

**외부 솔루션**: MongoDB, Redis, Pinecone

> ⚠️ **주의**: 벡터 데이터베이스 선택은 **RAG 성능**에 직접 영향을 미칩니다. 검색 속도와 정확도를 모두 고려해야 합니다.

## Bedrock 에이전트(Agents): 자동화된 의사결정

### 에이전트가 하는 일

**에이전트**는 단순히 질문에 답하는 것을 넘어 **자동으로 행동**합니다:

✅ 고객 구매 이력 조회 → 추천 → 자동 주문 생성
✅ 인프라 상태 확인 → 문제 감지 → 자동 복구 스크립트 실행
✅ API 호출 → 데이터 처리 → 결과 종합

### 에이전트 아키텍처 (사고 연쇄 방식)

```mermaid
flowchart TD
    A[사용자 요청] --> B[Bedrock 모델]
    B --> C[행동 계획 생성]
    C --> D[단계별 실행]
    D --> E{더 실행할 단계?}
    E -->|Yes| F[API/Lambda 호출]
    F --> D
    E -->|No| G[최종 응답 생성]
    G --> H[사용자에게 반환]
```

**에이전트가 수행할 수 있는 작업**: 외부 API 호출, AWS Lambda 함수 실행, 데이터베이스 조회/업데이트, Knowledge Base 검색, 순차적 다단계 작업

> 💡 **시험 포인트**: 에이전트는 '사람 개입 최소화'와 '복잡한 워크플로우 자동화'가 핵심입니다.

## 가드레일(Guardrails): 안전성과 규정 준수

### 가드레일의 역할

**가드레일**은 사용자 요청과 모델 응답을 **여러 층에서 필터링**합니다.

### 주요 기능

✅ **유해 콘텐츠 필터링**: 부적절한 요청 차단
✅ **PII 제거**: 이름, 이메일, 전화번호, 신용카드 번호 등 개인정보 마스킹
✅ **할루시네이션 감소**: 사실 기반 응답 강제
✅ **규정 준수**: HIPAA, PCI-DSS 등 산업 규정 준수

### 다층 방어 전략

| 계층 | 역할 | 예시 |
|------|------|------|
| 입력층 | 요청 사전 검증 | PII 탐지, 프롬프트 인젝션 방어 |
| 처리층 | 모델 행동 제어 | 응답 토큰 제한, 콘텐츠 범위 제한 |
| 출력층 | 응답 정화 | 민감 정보 마스킹, 사실성 검증 |

> 💡 **시험 포인트**: 가드레일은 '보안', '프라이버시', '규정 준수'의 삼각형을 이룹니다. 각 요소의 대표 예시를 알아두세요.

## 모델 평가(Evaluation)

### 두 가지 평가 방식

#### 1. 자동 평가 (Automatic Evaluation)

- 벤치마크 질문 & 정답 제시
- 모델이 답변 생성
- 심판 AI 모델이 BERTScore, F1 스코어 등으로 자동 채점
- 지원 작업: 텍스트 요약, 질문 답변, 텍스트 분류, 개방형 텍스트 생성

#### 2. 인적 평가 (Human Evaluation)

- 전문가(SME) 또는 내부 팀이 벤치마크 구성
- 모델 응답을 직접 비교 채점 (엄지 위/아래, 순위 등)
- 비즈니스 기준 반영 가능 — 미묘한 품질 차이 포착

### 비즈니스 평가 메트릭

| 메트릭 | 측정 대상 |
|--------|----------|
| 사용자 만족도 | 응답에 대한 피드백 조사 |
| 전환율 | 생성 AI 기능 통한 구매 등 목표 달성 |
| ARPU | 사용자당 평균 수익 |
| 효율성 | 계산 리소스 사용량 |

## 모니터링과 로깅 (Amazon CloudWatch 통합)

Amazon Bedrock은 **모든 모델 호출**을 Amazon CloudWatch에 기록합니다.

✅ **모델 호출 로깅**: 입력/출력 텍스트, 이미지, 임베딩 → CloudWatch Logs 또는 Amazon S3 저장
✅ **CloudWatch Metrics**: 사용량, 콘텐츠 필터링 횟수 등 지표 자동 수집
✅ **CloudWatch Alarms**: 임계값 초과 시 자동 알림 (예: 가드레일 차단 급증)

> 💡 **시험 포인트**: 로깅은 '운영 모니터링', '규정 준수 추적', '보안 감시'의 핵심 수단입니다.

## 가격 모델 완벽 가이드

### 세 가지 가격 옵션 비교

| 옵션 | 특징 | 할인 | 적합 워크로드 |
|------|------|------|----------|
| 온디맨드 | 토큰 단위 과금, 약정 없음 | 없음 | 예측 불가능, 가변적 |
| 배치 처리 | 다수 요청 일괄 처리, 비실시간 | **최대 50%** | 야간 배치, 대량 처리 |
| 프로비저닝 처리량 | 월/6개월 약정, 처리량 보장 | 장기 계약 할인 | 대규모 안정적 운영 |

**파인튜닝/커스텀 모델**은 반드시 **프로비저닝 처리량**을 사용해야 합니다 (온디맨드 불가).

### 비용 최적화 전략

| 전략 | 비용 영향 | 방법 |
|------|---------|------|
| 프롬프트 엔지니어링 | 매우 낮음 | 불필요한 토큰 제거 |
| RAG 활용 | 중간 | 재학습 대신 검색으로 대체 |
| 명령어 기반 파인튜닝 | 중간 | 레이블 데이터 활용 |
| 전체 파인튜닝 | 높음 | 비레이블 대규모 데이터 |

> ⚠️ **주의**: Temperature, Top K, Top P 등 **파라미터 조정은 비용에 영향 없음** — 모델 동작만 바뀝니다.

> 💡 **시험 포인트**: '**배치 모드 = 최대 50% 할인**', '**커스텀 모델 = 프로비저닝 필수**' — 반드시 암기하세요.

## 전이 학습(Transfer Learning)

파운데이션 모델은 이미 대규모 데이터로 사전학습되어 있습니다. 이 **기초 지식을 활용**하여 새로운 작업에 빠르게 적응하는 것이 전이 학습이며, **파인튜닝은 전이 학습의 한 종류**입니다.

**예시**:
- 자동차 이미지 분류 사전학습 → 강아지 품종 분류로 전이
- 언어 이해 사전학습 → AWS 기술 문서 분류로 전이

## 예제 문제

**문제**: 한 회사가 고객 서비스 챗봇에 회사 내부 제품 문서를 기반으로 답변을 제공하려 합니다. 문서는 매주 업데이트됩니다. 가장 적합한 접근 방식은?

A) 파운데이션 모델을 매주 새롭게 파인튜닝한다
B) Amazon Bedrock Knowledge Bases와 RAG를 사용하여 최신 문서를 동적으로 검색한다
C) 온디맨드 모드 대신 프로비저닝 처리량으로 전환한다
D) 더 큰 컨텍스트 윈도우를 가진 모델로 교체한다

> **정답: B** — 매주 업데이트되는 데이터는 RAG가 최적입니다. 파인튜닝은 재학습이 필요하여 비용·시간이 과다 소요됩니다.

## 학습 체크리스트

AIF-C01 시험 대비 자체 점검 목록:

- [ ] Bedrock의 '완전 관리형' 의미와 데이터 프라이버시 보장 방식 이해
- [ ] 파운데이션 모델 4가지(Titan, Claude, Llama-2, Stable Diffusion) 특징 및 선택 기준
- [ ] 파인튜닝 3가지 방식 비교: 명령어 기반 vs 지속적 사전학습 vs 대화형
- [ ] RAG가 필요한 상황 vs 파인튜닝이 필요한 상황 구분
- [ ] 벡터 데이터베이스 4가지(Amazon OpenSearch Service, Amazon Aurora, Amazon Neptune Analytics, Amazon S3 Vectors)
- [ ] 에이전트의 사고 연쇄 방식(Chain of Thought) 과정 6단계
- [ ] 가드레일의 3층 방어(입력·처리·출력) 각 역할
- [ ] 자동 평가 vs 인적 평가 차이점 및 적합 상황
- [ ] Amazon CloudWatch 통합 3가지 기능(로깅, 메트릭, 알람)
- [ ] 가격 모델 3가지 비교 및 50% 할인 조건
- [ ] 커스텀/파인튜닝 모델에 프로비저닝 처리량이 필수인 이유

## 결론

Amazon Bedrock은 AWS 생성형 AI 서비스의 **핵심 기반**입니다. AIF-C01 시험에서 Bedrock 관련 문제는 반드시 출제됩니다.

**핵심 공식**:
- '자주 변하는 데이터' → **RAG**
- '말투·스타일 변경 또는 도메인 전문화' → **파인튜닝**
- '비실시간 대량 처리' → **배치 모드 (-50%)**
- '커스텀 모델 배포' → **프로비저닝 처리량 필수**

이 네 가지 공식만 완벽히 이해해도 Bedrock 관련 문제의 대부분을 풀 수 있습니다!
