# 재설계 규칙 (11개)

영문 AWS 시험 문제를 한국어 4지선다 문제로 재설계할 때 반드시 준수해야 하는 규칙입니다.

---

## 규칙 0 (최우선): 시나리오는 원문과 완전히 달라야 한다

원문 업종·상황·배경을 그대로 번역하는 것은 저작권 침해이므로 절대 금지한다.

핵심 AWS 개념(정답 서비스와 이유)은 유지하되, **업종·배경·등장인물·문제 상황은 완전히 새로 창작**한다.

### 시나리오 변환 예시

| 원문 시나리오 | 재설계 시나리오 (예시) |
|---|---|
| 소셜미디어 댓글 유해언어 탐지 | 게임사 멀티플레이어 채팅 필터링 |
| 뉴스 기사 감성 분석 | 이커머스 상품 리뷰 품질 분류 |
| 의료 기록 개체명 추출 | 물류 계약서 주요 조항 추출 |
| 영문 제품 설명 다국어 번역 | 글로벌 HR 정책 문서 현지화 |

### 창작 가이드

1. 원문 업종이 "소셜미디어"라면 → 게임, 금융, 헬스케어, 제조, 물류 등 **다른 업종**으로 변경
2. 원문 문제 상황을 **핵심 기술 요구사항**으로만 추상화한 뒤, 새 맥락에서 재서술
3. 등장 회사 유형은 `규칙 3`의 글로벌 중립 기업 목록에서 선택

### 자가 검증

재설계 완료 후 다음 질문에 "아니오"가 나와야 통과:
- "원문과 같은 업종인가?" → 아니오여야 함
- "원문 시나리오 문장을 번역한 수준인가?" → 아니오여야 함

## 규칙 1: AWS 서비스명 원문 그대로 보존

원문에 등장한 AWS 서비스명은 번역·축약·대체 없이 반드시 그대로 사용한다.

- ✅ Amazon SageMaker → Amazon SageMaker
- ✅ Amazon S3 → Amazon S3
- ❌ Amazon SageMaker → "ML 관리형 서비스"
- ❌ Amazon Bedrock → "생성형 AI 플랫폼"

원문에 없던 AWS 서비스를 보기에 추가하는 것은 허용되나, 원문 서비스명이 변경되어서는 안 된다.

---

## 규칙 2: 자연스러운 한국어 작성 (번역투 금지)

직역체·번역투 표현 대신 자연스러운 한국어 수험 문체를 사용한다.

- ❌ "당신은 어떤 서비스를 사용하시겠습니까?"
- ✅ "다음 중 가장 적합한 AWS 서비스는 무엇인가?"
- ❌ "이 회사는 원한다 to..."
- ✅ "이 회사는 ...하고자 합니다."

조사, 어미, 경어체(-입니다/-합니다)를 일관되게 사용한다.

---

## 규칙 3: 글로벌 중립적 시나리오 사용

지역·국가를 특정하는 표현을 사용하지 않는다. 언어는 한국어이지만 배경은 글로벌 중립적이어야 한다.

- ✅ "한 글로벌 핀테크 기업의 ML 엔지니어가..."
- ✅ "한 대형 소매업체의 데이터 사이언티스트가..."
- ❌ "서울에 본사를 둔 한국 기업이..."
- ❌ "한 한국 스타트업이..."

회사 유형 예시: 글로벌 테크 기업, 금융 서비스 회사, 헬스케어 기업, 전자상거래 플랫폼, 미디어 스트리밍 서비스, 자동차 제조사, 물류 회사.

## 규칙 4: 보기 길이 균형 유지 및 형식 단순화

보기 텍스트는 **짧고 명확하게** 작성하며, 4개 보기의 길이와 형식을 통일한다.

### 보기 형식 우선순위 (권장 → 비권장)

| 우선순위 | 형식 | 예시 |
|---------|------|------|
| ✅ 1순위 | 서비스명 단독 | `Amazon Textract` |
| ✅ 2순위 | 서비스 조합 (`A + B`) | `Amazon Transcribe + Amazon Comprehend` |
| ✅ 3순위 | 간결한 명사구 | `모델 재학습 (Re-training)` |
| ⚠️ 허용 | 짧은 서술구 | `SageMaker Feature Store를 사용하여 피처를 저장한다` |
| ❌ 금지 | 파이프 구분 복합 매핑 | `요구 1: SageMaker A \| 요구 2: SageMaker B \| 요구 3: ...` |
| ❌ 금지 | 한 보기 안에 여러 서비스 열거 | `SageMaker + S3 + Lambda + DynamoDB` |

### 구체적 기준

- **최대 권장 길이:** 보기 1개당 40자 이내 (부득이한 경우 60자까지)
- 4개 보기의 텍스트 길이 차이: 최대 50% 이내
- 모든 보기를 **같은 문법 구조**로 통일 (명사구 또는 서비스명)
- 정답에만 부연 설명을 넣어 길게 만들지 않는다

### ❌ 나쁜 예 vs ✅ 좋은 예

```
❌ 나쁜 예 (파이프 복합 매핑 — 읽기 불편, 4가지를 동시 비교 불가)
A. 요구 1: SageMaker Studio | 요구 2: SageMaker Experiments | 요구 3: AWS Step Functions | 요구 4: S3
B. 요구 1: SageMaker Data Wrangler | 요구 2: SageMaker Feature Store | 요구 3: SageMaker Pipelines | 요구 4: SageMaker Model Registry

✅ 좋은 예 (서비스명 단독 — 즉시 비교 가능)
A. Amazon Transcribe
B. Amazon Textract
C. Amazon Comprehend
D. Amazon Rekognition

✅ 좋은 예 (서비스 조합 — 2개까지 허용)
A. Amazon Polly + Amazon Rekognition
B. Amazon Transcribe + Amazon Comprehend
C. Amazon Lex + Amazon Translate
D. Amazon Textract + Amazon Kendra
```

## 규칙 5: 독립적으로 그럴듯한 오답 3개 구성

각 오답은 다음 기준을 충족해야 한다:

- 관련 AWS 서비스나 개념을 포함할 것
- 수험생이 실제로 고민할 수 있을 만큼 그럴듯할 것
- 정답과 완전히 무관한 엉터리 답안은 사용하지 않을 것
- 오답끼리도 서로 다른 개념을 다룰 것 (오답 2개가 거의 같은 말이면 안 됨)

## 규칙 6: 보기 간 중복/혼동 방지

두 보기가 사실상 같은 내용을 다른 말로 표현하지 않도록 한다.

- ❌ 보기 A: "Amazon S3에 데이터를 저장한다" / 보기 C: "S3 버킷을 사용하여 데이터를 유지한다"
- ✅ 각 보기는 서로 다른 서비스나 접근 방식을 나타낼 것

## 규칙 7: 반드시 단일 정답 (4지선다)

출력 문제는 반드시 정답이 1개인 4지선다 형식이어야 한다.

- 원문이 복수 정답(Choose TWO 등)이면 두 정답 개념을 통합하여 단일 정답 보기로 재설계
- 원문 보기가 5개 이상이면 가장 약한 오답을 제거하여 4개로 축소
- `correct_option_id`는 `'a'`/`'b'`/`'c'`/`'d'` 중 하나만 가능

## 규칙 8: 시나리오 + 명확한 질문 구조

문제는 두 부분으로 구성한다:

1. **시나리오/배경** (1~4문장): 회사 상황, 요구사항, 제약조건 설명
2. **질문** (1~2문장): "다음 중 ... 가장 적합한 것은?", "... 올바른 것은?" 등 명확한 질문

### 문제 본문 줄바꿈 서식 규칙

읽기 편의성을 위해 아래 규칙을 반드시 적용한다:

- **마지막 문장(질문 문장) 앞에 항상 빈 줄 삽입** (전체 문장 수와 무관하게)
- **전체 문장 수가 4개 이상이면 첫 번째 문장 다음에도 빈 줄 삽입**

> ⚠️ **JSON `text` 필드 작성 시 빈 줄은 반드시 `\n\n`(줄바꿈 2개)으로 표기한다.**
> - 3문장: `"text": "첫 번째 문장. 두 번째 문장.\n\n질문은?"`
> - 4문장+: `"text": "첫 번째 문장.\n\n두 번째 문장. 세 번째 문장.\n\n질문은?"`

**3문장 예시 (시나리오 2문장 + 질문 1문장):**
```
한 글로벌 미디어 기업이 사용자 행동 데이터를 분석하여 개인화된 콘텐츠 추천 시스템을 구축하려 합니다.
분석 팀은 데이터 준비부터 모델 훈련, 배포까지 단일 통합 환경에서 관리하고자 합니다.

다음 중 이 요구사항을 가장 효과적으로 충족하는 AWS 서비스는 무엇인가?
```
JSON: `"text": "...추천 시스템을 구축하려 합니다.\n분석 팀은...관리하고자 합니다.\n\n다음 중 이 요구사항을 가장 효과적으로 충족하는 AWS 서비스는 무엇인가?"`

**4문장 이상 예시 (시나리오 3문장 + 질문 1문장):**
```
한 글로벌 핀테크 기업이 신용 리스크 평가 모델을 자동화하려 합니다.

현재 팀은 수백만 건의 거래 데이터를 수작업으로 분석하고 있으며, 모델 재학습 주기가 너무 길어 최신 패턴을 반영하지 못하고 있습니다.
운영 환경에서 실시간 예측이 가능해야 하며, 인프라 관리 부담은 최소화해야 합니다.

다음 중 이 요구사항을 충족하는 가장 적합한 AWS 서비스는 무엇인가?
```
JSON: `"text": "...자동화하려 합니다.\n\n현재 팀은...반영하지 못하고 있습니다.\n운영 환경에서...최소화해야 합니다.\n\n다음 중 이 요구사항을 충족하는 가장 적합한 AWS 서비스는 무엇인가?"`

## 규칙 9: key_points 형식

핵심 개념을 다음 형식으로 작성한다:

```
{핵심 개념 제목}
• {포인트 1}
• {포인트 2}
• {포인트 3}
• {포인트 4 (선택)}
• {포인트 5 (선택)}
```

- 제목은 AWS 서비스명 또는 핵심 개념명으로 시작
- 포인트는 3개 이상 5개 이하
- 각 포인트는 한 줄에 핵심만 간결하게 작성
- `key_point_images` 필드는 사용하지 않음 (DB에 컬럼 없음)

## 규칙 11: 한 문제에 하나의 개념만 테스트

한 문제는 **단 하나의 핵심 개념·판단**을 물어야 한다. 여러 서비스를 여러 요구사항에 동시에 매핑하는 구조는 만들지 않는다.

### ❌ 금지: 다중 매핑 구조

```
요구사항이 4개 있고, 각각에 맞는 서비스를 보기 A~D에서 동시에 선택하는 문제
→ 수험생이 4개 판단을 동시에 수행해야 하므로 인지 부하 과다
→ 파이프 구분 보기로 이어지는 원인
```

### ✅ 권장: 단일 판단 구조

원문이 "4가지 요구사항 → 4개 서비스 매핑" 형식이라면, 핵심 요구사항 **1개**를 선택하여 "이 요구사항에 맞는 서비스는?" 형식으로 재설계한다.

```
원문: 요구 1(데이터 전처리)→Wrangler, 요구 2(피처 관리)→FeatureStore, 요구 3(파이프라인)→Pipelines, 요구 4(모델 관리)→ModelRegistry
재설계: "훈련에 사용한 피처를 버전 관리하고 온라인/오프라인으로 서빙하려 할 때 적합한 서비스는?"
→ A. SageMaker Data Wrangler / B. SageMaker Feature Store / C. SageMaker Pipelines / D. SageMaker Model Registry
```

## 규칙 12 (저작권 보호): 보기 순서 무작위화 및 정답 위치 변경 필수

재설계 완료 후 보기 순서를 반드시 원문과 다르게 배치한다.

### 처리 절차

1. 원문 `answer` 필드에서 정답 위치 확인 (예: `"a"` → 원문 정답이 a 위치)
2. 재설계된 정답 보기를 원문 정답 위치가 **아닌** 다른 위치(b/c/d 중 하나)에 배치
3. 오답 3개도 나머지 위치에 원문과 다른 순서로 배치
4. `correct_option_id`와 `sort_order`를 새 배치에 맞게 업데이트

### 예시

```
원문 정답: A (Amazon Comprehend Medical)

재설계 전 (❌ 원문과 동일한 순서):
a. Amazon Comprehend Medical ← 정답, 원문과 같은 위치
b. Amazon Personalize
c. Amazon Textract
d. Amazon Kendra

재설계 후 (✅ 순서 변경):
a. Amazon Personalize
b. Amazon Textract
c. Amazon Comprehend Medical ← 정답, 원문과 다른 위치
d. Amazon Kendra
```

### 자가 검증

- "재설계 `correct_option_id` == 원문 `answer`?" → **NO**여야 통과

## 규칙 10: ref_links 형식

AWS 공식 문서 링크를 JSON 배열로 1~3개 제공한다.

```json
[
  {"name": "문서 제목", "url": "https://docs.aws.amazon.com/..."},
  {"name": "문서 제목 2", "url": "https://aws.amazon.com/..."}
]
```

- 허용 도메인: `docs.aws.amazon.com`, `aws.amazon.com`
- URL은 LLM이 자동 생성 (hallucination 허용, 사후 검토 방식)
- 관련 서비스 공식 문서 페이지를 우선적으로 참조
